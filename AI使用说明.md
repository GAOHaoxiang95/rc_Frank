# AI 使用说明

这个项目在需求分析和方案设计阶段我用了 AI 来帮忙。AI 给的第一版方案我原样留在了
[`ai-design-initial-version.md`](./ai-design-initial-version.md) 里没动，方便对比。

---

## 一、AI 帮到我的地方

- **帮我把架子先搭起来了**。它一上来就给了"接入层 / 存储层 / 调度投递 / 失败重试 / 查询"这套分层，
  还有 `notification_task` + `delivery_attempt` 两张表的雏形。省得我从零去列模块，我直接在它基础上
  加加减减就行。

- **"先别上 MQ"这个方向它和我想到一块了**。它建议用"数据库任务表 + worker 轮询"来做可靠投递，
  而不是一上来就整个消息队列。这个我很认同，符合"别盲目堆复杂度"的思路，我后面只是把"为什么这么选"
  讲得更透了一点。

- **系统边界的初稿是它列的**。哪些做、哪些不做（异步投递、持久化、重试、不保证 exactly-once、
  不做模板编排/流程引擎/多租户），它给了个能直接讨论的起点，我在上面改了改。

- **演进路线给了我一堆备选项**。轮询换 MQ、加死信队列、模板配置、多实例分片、限流熔断、敏感字段加密
  这些，当"以后怎么扩"的素材挺好用，我挑了合理的留下。

- **测试要覆盖哪些场景它也帮我想全了**。用 WireMock/MockWebServer 去覆盖成功、超时、5xx 重试、
  4xx 失败、重启续投、幂等提交这些，清单挺全的，基本照用。

---

## 二、AI 的建议里，我没采纳 / 觉得它没做到位的

这一节顺便也把**第一版 AI 方案的不足**集中说一下。

### 2.1 我觉得"没必要做那么多"的部分

先说清楚：AI 这第一版其实挺克制的，它自己就主动把 MQ、模板编排、流程引擎、多租户都排除在 MVP 外了
（原文写的是"MVP 不直接引入 MQ"、"不做复杂供应商模板编排、流程引擎、多租户权限体系"）。这点我跟它想法
一致，谈不上"我推翻它"，更多是认同了它的克制。

我真正做减法的，是它**演进路线**里那一长串东西：

- **演进路线那串我不会当成"迟早都得做"的待办**。Kafka/RabbitMQ、死信队列+管理后台、模板配置、
  多实例分片、限流熔断、字段加密……列着没问题，但我不打算挨个排期。每一项我都绑了触发条件才做，
  比如 MQ 和分片得等到"DB 轮询真的开始积压、worker 加机器也顶不住 DB"了才考虑；管理后台得等到
  人工介入频繁到值得做个界面了才做。说白了就是别为还没到的规模提前还债。

- **供应商模板编排 / 流程引擎不做**。这块 AI 本来也归到"不解决"里了，我认同，并且补了下"为什么不做"
  （详见第三节第 3 点）——硬要平台去适配每家供应商的格式，复杂度只会越堆越高，这活儿更适合调用方自己干。

- **完整多租户那套也不做**。AI 也没要做，我同意。第一版面向公司内部系统，不需要做租户隔离、租户计费、
  租户级权限模型，但这不等于默认所有内部调用都可信；调用方身份、source system 白名单、目标 URL 白名单
  这些基础控制还是要有。

### 2.2 AI 第一版漏掉 / 答得不够的地方（这些是它的短板）

- **完全没提 SSRF 这个安全坑**。这服务本质就是"你给我个 `targetUrl`，我替你去请求"，AI 却没做任何
  URL 校验、内网网段拦截、协议限制、调用方鉴权。这是最要命的漏的——一不小心这服务就成了别人探内网的
  跳板。所以我把"出网安全（只允许 https + 内网网段拉黑 + 调用方鉴权）"直接提成第一版必做。

- **PROCESSING 状态会卡死，它没考虑**。任务被 worker 锁成 PROCESSING 之后，要是 worker 崩了或者重启了，
  这条任务就**永远卡在 PROCESSING**，再也没人捞它。AI 只保证了 PENDING 重启后能接着投，PROCESSING 没管。
  我加了个 `locked_until` 租约 + 超时回收。

- **多实例并发它推到"以后再说"**。它把"多实例加锁"放演进路线了，意思是第一版你只要起两个实例，
  就会两个 worker 抢同一条任务、重复投递。其实 `SELECT ... FOR UPDATE SKIP LOCKED` 成本很低，
  我把它拉到第一版就做。

- **重试策略写得太糙**。它说"4xx 默认不重试"，但 `429`（被限流）和 `408` 也是 4xx，这俩恰恰得重试，
  还得看 `Retry-After`；而且退避没加 jitter，一批任务一起失败就会一起重试，容易把对方打挂。这块我改了。

- **幂等只防住了"上游重复提交"，没管"下游重复投递"**。它的 `idempotency_key` 只用来防业务方重复建任务。
  但你都选 at-least-once 了，对外部系统重复请求是躲不掉的，它却没说怎么帮下游去重。我补了一条：
  把幂等键透传到下游请求头里，让对方有机会自己去重。

- **数据会一直涨，它没给对策**。`delivery_attempt` 每投一次就一行，成功的任务还永久留着，没归档、没 TTL、
  没分区，跑久了表就爆了。我补了清理策略。

- **任务最终失败了没人知道**。它对"长期不可用"的回答就停在"进 FAILED + 提供手动 retry 接口"，没有告警、
  没有积压可见性。可没人去盯 FAILED 表，那不就等于没处理嘛。我补了几个关键指标（积压量、失败率、投递延迟）
  和告警思路。

- **`callback` 字段边界不清楚**。它 DTO 里有个可选 `callback`，全文没说这玩意儿用来干啥：
  是回调外部 API 的原始响应、通知调用方最终状态，还是失败补偿入口？需求里说业务方不需要同步关心外部
  API 的返回值，第一版如果再加 callback，相当于又多了一条"回调调用方"的可靠投递链路，也要重试、
  鉴权、限流和状态记录。我把它删掉，第一版改成调用方拿 `notificationId` 主动查询；以后真需要再单独
  设计 `completionWebhookUrl` 这类明确语义的终态通知能力。

## 三、哪些关键决定是我自己拍的，为何这么拍

1. **投递语义定成 at-least-once，并且说清楚为何不是 exactly-once。**
   外部系统不归我管，只要我会重试 + 网络又不靠谱，重复就消不掉——响应丢了的时候，我根本分不清是
   "请求没送到"还是"送到了只是回执丢了"，想保证送达就只能接受可能重复。真正的 exactly-once 得收发双方
   一起搞去重和确认协议，可供应商是第三方、连返回值我都拿不到，根本没条件做。所以我的做法是：
   **服务自己这边用状态机、唯一约束、租约和任务锁尽量减少重复处理，但不承诺 exactly-once；
   对外明确承诺 at-least-once，再把幂等键透传给下游，让下游有机会按业务语义去重。**
   这里还有一层因果：正因为连我自己这侧都消不掉重复（发完请求、状态还没落库就崩了，恢复后照样重投），
   "把幂等键透传给下游"就不是锦上添花，而是 at-least-once 语义下**必须**给下游留的去重抓手。

2. **第一版就死磕"数据库任务表 + 应用内 `@Scheduled` 轮询 worker"，不上 MQ。**
   理由前面说过：在这个作业和 MVP 假设的低到中等吞吐下，DB 方案足够把可靠投递闭环跑通，而且 DB 顺带
   把持久化、状态可查、事务一致性都解决了；MQ 的好处得等高吞吐、高扇出才显出来。我在设计文档里也写明了
   "啥时候该换 MQ"（积压一直涨、投递延迟扛不住、worker 加机器被 DB 锁等待或扫描成本卡住）。

3. **供应商适配的活儿交给调用方，平台只负责"老老实实把它投出去"。**
   需求里强调每家供应商的 URL/Header/Body 都不一样。我的决定是：平台收调用方拼好的
   `targetUrl/method/headers/body`，**自己不做模板编排**。因为模板这东西会随供应商数量一直膨胀成平台的
   负担，而且格式本来就是业务自己最清楚的事。代价（每个业务系统各自适配）我在文档里也老实写了。
   如果以后出现多个业务系统重复对接同一供应商，或者签名、鉴权、Body 映射开始大量复制，再把"统一模板配置"
   或 connector 抽到平台侧。

4. **安全（SSRF 防护 + 调用方鉴权）提到第一版必做**，而不是像 AI 那样放到"以后再加密脱敏"。
   原因很简单：一个能替你请求任意 URL 的服务，安全就是正确性的一部分，不能等出事了再补。即使这是公司
   内部几个系统在调，也不能默认全可信；第一版至少要有调用方身份、source system 白名单、目标 URL 白名单
   或出网策略，以及基础限流。

5. **并发正确性（SKIP LOCKED 领任务 + PROCESSING 租约回收）也提到第一版。**
   "可靠投递"这四个字，首先得保证它自己在多实例、崩溃重启的情况下不丢不卡，这是地基，不是优化项。

6. **接收契约说明白。** 接口保证"落库成功才返回 notificationId"，并跟调用方约好：超时或者没拿到 ID，
   就带同一个 `idempotencyKey` 重试。这样可靠性的边界对调用方来说清清楚楚、能依赖。

7. **调度就用应用内 `@Scheduled`，没上 Quartz 这类集群调度。**
   AI 的第一版只说了"worker 轮询"，没明确调度触发器。我一度考虑过 Quartz JDBC JobStore（图它的集群协调），
   但想清楚后否掉了：dispatcher 的并发安全已经被 `SKIP LOCKED + locked_until` 兜住——**就算每个实例都各自
   `@Scheduled` 同时跑，也不会重复领取同一条任务**，根本不需要 Quartz 那种"同一时刻只让一个节点跑"的协调。
   为这点引入 Quartz 的额外表、配置和运维，属于过度设计。所以最终：dispatcher/reaper 用
   `@Scheduled`，通知任务放 `notification_task`，靠 `next_attempt_at + SKIP LOCKED + locked_until` 做领取、
   重试和崩溃恢复，第一版不引入任何额外调度中间件。

---

> 总结：AI 帮我把**架子和备选项**快速搭出来了，但在**安全、并发正确性、失败可见性**这些靠真实
> 工程经验才想得到的地方，要么漏了要么过了。我主要干的事，是把它的输出**收边界、补风险、把取舍讲清楚**，
> 而不是照单全收。
